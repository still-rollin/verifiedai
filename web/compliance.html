Compliance logic is rules over numbers: thresholds, slabs, exemptions. Rules can be theorems. Theorems can be machine-checked. And mis-stated rules can be caught — before they ship inside software that moves money.

λThe formalizer

Two of the three modules are not written by hand. A small compiler (tools/formalize_tax.py, tools/formalize_tds.py) takes the published rule table as input and emits Lean. No LLM: the table is the spec, the generator is a compiler, the kernel checks the output.

# input — the new-regime slab table, FY 2024-25
SLABS = [(300_000, 0), (700_000, 5), (1_000_000, 10),
         (1_200_000, 15), (1_500_000, 20), (None, 30)]
-- output (excerpt) — Compliance/IncomeTax.lean, generated
def slabTax (i : Nat) : Nat :=
    if i ≤ 300000 then 0
    else if i ≤ 700000 then 0 + (i - 300000) * 5 / 100
    else if i ≤ 1000000 then 20000 + (i - 700000) * 10 / 100
    …

-- every slab boundary, checked by the kernel
example : slabTax 700000 = 20000   := by decide
example : slabTax 1000000 = 50000  := by decide

-- §87A rebate: a real theorem, with a hypothesis
theorem nr25_rebate_nils_tax (i : Nat) (h : i ≤ 1200000) :
    nr25_payable i = 0 := by simp [nr25_payable, h]

-- marginal relief, proved: ₹1 above the limit costs ₹1, not ₹60,000
theorem nr25_marginal_relief (i : Nat) (h : 1200000 < i) :
    nr25_payable i ≤ i - 1200000 := …
example : nr25_payable 1200001 = 1 := by decide

The TDS module is generated the same way from a 20-section threshold table: constants, deductibility predicates, boundary checks at the threshold and ₹1 above, and a monotonicity theorem per section. Adding a section is one line in the table; the proofs come out of the compiler.

⊥The audit — two planted defects, both caught

The GST module is hand-written: registration above ₹40 lakh, composition eligibility up to ₹1.5 crore, e-invoicing from ₹5 crore, with two proved relations between them. It also contains two rules mis-formalized the way real mistakes happen. The auditor's output:

The vacuous one mixed the ₹20 lakh services threshold with the ₹40 lakh goods threshold. Its hypotheses contradict each other, so the "exemption rule" exempts nobody — and every proof of it is worthless. The trivial one compares two constants where it meant to talk about a business's turnover. Both compile. Both look certified. Both say nothing. That is the failure mode the auditor exists to catch, and it is enforced as a regression test: exactly these two flag, everything real stays clean.

≈Faithfulness — what machines can and cannot check

"Does the formal rule match the law?" cannot be machine-proved: the law is an informal artifact. Our design answers it in three layers. First, the rules are generated from the published rate tables, so the only thing requiring human review is the table itself — forty numbers a chartered accountant can check against the Finance Act in minutes, instead of pages of formal code. Second, the auditor mechanically eliminates statements that are secretly meaningless regardless of the numbers. Third, a reference-vector module: independently published worked examples — the Budget 2025 marginal-relief illustrations, standard CA references — encoded as kernel-checked facts against our generated rules:

-- Budget 2025 illustration: ₹12,10,000 → slab ₹61,500, payable ₹10,000
example : nr25_slabTax 1210000 = 61500 := by decide
example : nr25_payable 1210000 = 10000 := by decide
-- ₹25,00,000 → the widely-quoted ₹3,30,000
example : nr25_payable 2500000 = 330000 := by decide

Fourteen such vectors, all three regimes. For our table to be wrong, its error would have to reproduce the government's own published arithmetic — the failure modes are independent, which is the point of a cross-check.

∀Why this domain

Tax engines, GST filing software and payroll systems encode these rules today as untested conditionals. When a threshold is wrong, the error surfaces as a notice, a penalty, or a refund that never comes. Formalized rules give three things: the rule table is the single source of truth, every boundary case is kernel-checked, and — the part nobody else does — the statements themselves are audited, so a wrong formalization cannot hide behind a valid proof.

v1.0 says what it omits: surcharge, deductions and exemptions, capital-gains rates, PAN rules, and month-level aggregation windows. Cess and §87A marginal relief are in. Extending coverage is table entry, not research — the two compilers do the proofs.
