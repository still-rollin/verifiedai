The kernel checks your proofs.
Nobody checks your statements.

A formally verified theorem is only as true as its statement — and in the field's flagship benchmark, over half of the failing problems turned out to be statement bugs that went unnoticed for five years. verifiedai is CI for formal proofs: it audits what your theorems actually say, and repairs your proofs when dependencies break them.

$ verifiedai audit ./project Demo/Pricing.lean

STATEMENT AUDIT

Every check is a real Lean compilation, not a heuristic: derive False from the hypotheses to catch vacuous theorems, probe for trivially-true statements, surface unused hypotheses. Machine-certified proofs of human-unchecked statements is the trust hole in formal verification — this closes it.

PROOF REPAIR

mathlib moves fast and every version bump breaks downstream proofs. verifiedai fixes them with a deterministic-first ladder — library search, tactic cascade, then an LLM — and only accepts fixes the Lean kernel verifies. It cannot ship a wrong fix.

RUNS IN YOUR CI

A drop-in GitHub Action: every pull request gets its new theorem statements audited and findings posted as a review comment. Dependabot keeps dependencies fresh; verifiedai keeps proofs honest.

Why now. AI labs are producing machine-checked mathematics at industrial scale, and the entire trust story rests on "the kernel verified it." But the kernel only checks the proof against the statement — miniF2F-v2 showed measured prover accuracy jumped from 40% to 70% once statement bugs were fixed. As formal verification goes mainstream, auditing statements becomes the missing layer of the stack.

How the audit works

No heuristics, no vibes: every finding is the result of a real Lean compilation. The auditor turns each theorem into adversarial probe programs and asks the kernel itself to judge them.

1

Parse the claim

Split every theorem into its hypotheses and its conclusion.

theorem big_discount_still_positive
  (p d : Nat) (h1 : d > p + 10) (h2 : d < p) :   -- hypotheses
  applyDiscount p d > 0                           -- conclusion
2

Generate adversarial probes

Write new Lean programs that attack the statement. The vacuity probe keeps the hypotheses and swaps the conclusion for False:

-- if this COMPILES, the theorem is about nothing
theorem __vac (p d : Nat)
  (h1 : d > p + 10) (h2 : d < p) : False := by
  first | omega | simp_all | decide
3

Let the kernel judge

Each probe is compiled against the real project. A probe that compiles is a mathematical fact about your statement, not an opinion:

probe compiles  →  ✗ finding (provable, not arguable)
probe fails     →  ✓ statement survives the attack

And repair? The inverse discipline. Candidate fixes come from a ladder — Lean's own library search (exact?), a deterministic tactic cascade, then an LLM with compile-error feedback — but a candidate only survives if the file recompiles under the kernel. The LLM proposes; the kernel disposes. A wrong fix cannot ship, by construction.

The check catalog

Statement trust is not one check — it's a growing battery of attacks. What runs today, and what the roadmap funds:

CheckWhat it catchesStatus
Vacuous hypotheses Contradictory assumptions — the theorem is true about nothing. The kernel happily accepts it; the vacuity probe derives False and exposes it. LIVE
Trivially-true statement The goal closes under rfl / omega / decide / simp alone — a strong smell that the formalization simplified away the actual claim. LIVE
Unused hypotheses Assumptions the proof never touches — over-constrained statements, or a claim quietly narrower than the informal one. LIVE
Proof repair Dependency bumps that break proofs — fixed by the exact? → cascade → LLM ladder, kernel-gated. LIVE
Back-translation faithfulness An LLM that never saw the informal statement translates the formal one back to English; disagreement with the original flags unfaithful formalization. Calibrating this against symbolic signals is an open research problem — and our next milestone. ROADMAP
Definition drift The classic cheat: redefine Prime locally, prove something easy about it. Detects statements bound to shadowed or non-standard definitions. ROADMAP
Spec mutation testing For verified code: inject known-buggy implementations — if the spec still verifies, the spec is too weak (the ensures true problem documented across vericoding benchmarks). ROADMAP
Benchmark audit mode Sweep an entire benchmark or proof corpus and produce a statement-integrity report — validated against miniF2F-v1's now-documented errors as ground truth. ROADMAP
Rocq/Coq support The same probe battery for the Rocq ecosystem, reusing our Rocqet/Oraqle infrastructure. ROADMAP

Who needs statement trust

AI-FOR-MATH LABS

Labs generating millions of proofs market them as "hallucination-free" — a claim that is only as strong as the statements those proofs attach to. An independent audit layer is what makes the claim survive scrutiny.

BENCHMARK MAINTAINERS

miniF2F mismeasured the entire field for five years; fixing its statements moved measured accuracy from 40% to 70%. Every eval built on formal statements needs a statement-integrity gate before results are trusted.

LIBRARY & PROJECT MAINTAINERS

Every mathlib release breaks downstream proofs, and every contributor PR can introduce a vacuous lemma that reviewers wave through. verifiedai runs in CI: repair on bump, audit on PR.

VERICODING TEAMS

LLMs writing verified code learn to game weak specs — ensures true passes every verifier. Spec-strength auditing is the difference between proven correct and proven nothing.

JOURNALS & ARTIFACT REVIEW

Papers increasingly ship formal supplements ("we verified Theorem 3 in Lean"). Reviewers check that it compiles — not what it says. A one-command audit gives artifact evaluation teeth.

AI AGENTS THEMSELVES

Agents that write proofs need a conscience in the loop: via MCP, an agent audits every theorem it just proved before reporting success — closing the loop between generation and trust.

MCP-native: give your AI agent a conscience

AI agents now write formal proofs at industrial scale — and an agent that can prove things needs a way to check what it proved. verifiedai ships as an MCP server, so any MCP client — Claude Code, Cursor, or your own agent loop — can audit every theorem it generates and repair every build it breaks, without leaving the conversation.

# register once
$ claude mcp add verifiedai -- env PYTHONPATH=~/verifiedai/cli python3 -m verifiedai.mcp_server

# then any agent can call:
audit_file(project_root, file)      → vacuity / triviality / unused-hypothesis findings, as JSON
repair_project(project_root)       → kernel-verified fixes + what still needs a human

The research behind it

verifiedai grows out of two years of formal-methods research and tooling — retrieval, cross-prover proof representation, and neural-symbolic search — built alongside the founder's work at INRIA/CNRS Paris.

ROCQET ● LIVE

Semantic search for Rocq/Coq. Natural-language queries over the Rocq stdlib, MathComp, GeoCoq, UniMath and HoTT — "commutativity of addition" finds the lemma, no library-specific name recall needed. Deployed, public, MCP-integrated.

rocqet.vercel.app ↗

ORAQLE · MCP

Prover-agnostic proof extraction. Drives Rocq/Coq's kernel to capture, for every lemma application, the resolved arguments after unification — what the proof actually did, not what the surface syntax says. An AI agent calls get_resolved_calls("lemma_3_7a") and gets:

lemma_betweennotequal    A B C H
lemma_betweennotequal    B C D H0
lemma_localextension     A C D H1 H2
lemma_congruencesymmetric C C E D H6
lemma_3_6a               A B C E H H5
lemma_extensionunique    B C D E H0 H8 H7

Live output over Euclid's Elements in GeoCoq. The representation layer behind cross-prover translation — Rocqet finds it, Oraqle explains it, verifiedai checks it.

PROOFFILTER

Neural-symbolic tactic filtering. Prunes candidate proof steps before they reach the Lean kernel: 11.8× solve-rate improvement on miniF2F, 76.7% of kernel calls saved, 3.35× faster time-to-proof.
