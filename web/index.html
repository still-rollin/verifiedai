Continuous integration for formal proofs · Lean 4

The kernel checks your proofs.
Nobody checks your statements.

A formally verified theorem is only as true as its statement. verifiedai audits what theorem statements actually say — every finding is a compiled Lean probe, so false positives are impossible — and repairs proofs broken by dependency bumps, keeping only fixes the kernel verifies.

Results on public corpora

We run the auditor where the field's trust actually lives: the benchmarks provers are measured on, and the corpora they train on. Same battery each time, raw JSON published.

corpuswhat it isauditedfound

The gradient is the point: maintained corpora are clean; static and auto-generated ones are not, and nothing in the ecosystem re-audits the static ones. A defective statement in a benchmark skews a score. The same statement in training data teaches a prover to exploit contradictions. That is a CI problem.

What it does

verifiedai audit flags theorems the kernel accepts but that don't say what you think: contradictory (vacuous) hypotheses, statements that close under one tactic with all hypotheses stripped, unused hypotheses, and trust-chain cheats such as smuggled axioms, hidden sorry, and native_decide.

verifiedai repair fixes proofs broken by dependency bumps with a deterministic-first ladder: library search (exact?), then a tactic cascade, then an LLM with compile-error feedback. A candidate fix ships only if the file recompiles under the kernel. A wrong fix cannot ship.

$ verifiedai audit ./project Demo/Pricing.lean

  ✓ discount_le_price (line 10)
  ✗ big_discount_still_positive (line 16)
      [vacuous] the hypotheses are contradictory — False is derivable
      from them alone; this theorem proves nothing about any real case
  ⚠ tax_identity (line 22)
      [trivial] the conclusion holds without the hypotheses
      [unused_hypothesis] hypothesis `h` is never used

  3 theorems audited, 2 flagged

$ verifiedai repair ./project

  ✓ fixed Demo/Refunds.lean theorem refund_comm
      via exact? → exact Nat.add_comm a b
  1/1 broken declarations repaired (kernel-verified)

Method

The auditor turns each theorem into adversarial probe programs and lets the Lean kernel judge them. A probe that compiles is a mathematical fact about the statement, not an opinion.

Parse the claim

Split every theorem into hypotheses and conclusion.

theorem big_discount_still_positive
  (p d : Nat) (h1 : d > p + 10)
  (h2 : d < p) :          -- hypotheses
  applyDiscount p d > 0   -- conclusion

Attack it

The vacuity probe keeps the hypotheses and swaps the conclusion for False; the triviality probe keeps the conclusion and strips the hypotheses.

-- if this compiles, the theorem is about nothing
theorem __vac (p d : Nat)
  (h1 : d > p + 10) (h2 : d < p) : False := by
  first | omega | simp_all | decide

Let the kernel judge

Probes are batched, at most 20 theorems per compile, and inserted inside each theorem's own namespace, so auditing costs about two compiles per file. Each probe carries a #print axioms sentinel: a finding requires positive confirmation, so if Lean aborts mid-file the engine fails safe rather than misreading silence.

The checks

checkwhat it catchesstatus
vacuous contradictory hypotheses — the theorem is true about nothing live
trivial conclusion closes under rfl / omega / decide / simp with all hypotheses stripped live
unused hypothesis assumptions the declaration never uses live
trust chain smuggled axioms, sorry hidden behind conjunctions or macros, native_decide — read from Lean's own axiom introspection; the reward-integrity gate for prover training live
faithfulness an LLM that never saw the informal statement back-translates the formal one; a separate judge compares. Warnings only beta
definition drift statements bound to shadowed or locally-redefined definitions roadmap
spec mutation for verified code: inject known-buggy implementations; if the spec still verifies, the spec is too weak roadmap
Rocq/Coq support the same battery for the Rocq ecosystem roadmap

False negatives are expected: the battery is a lower bound, and every check added widens it. The adversarial zoo, a corpus of kernel cheats the auditor must catch forever, is enforced in CI so the battery compounds instead of regressing. When our own engine had a soundness bug, we published it with the fix and its regression tests.

Integration

CLI verifiedai audit <project> --all — sweep a project; JSON for machines, markdown for PR comments
GitHub Action drop one workflow file into any Lean repo; every PR gets its new statements audited, findings posted as a review comment
Self-hosted API verifiedai serve — zero-dependency HTTP server for on-prem use; proof corpora never leave your machines
MCP server agents that write proofs audit them before reporting success: audit_file and repair_project, structured JSON

Who it's for

Labs training provers. A model optimizing "the kernel accepted it" learns to smuggle axioms and mine contradictions — and its training data already contains both (see the Lean Workbook audit). verifiedai gates the reward signal and audits the corpus before the run, not in the postmortem.

Benchmark and dataset maintainers. Fixing miniF2F's statements moved measured prover accuracy from 40% to 70% (arXiv:2511.03108). A statement-integrity sweep is one command and about an hour per thousand statements, on a laptop.

Lean project maintainers. Every mathlib bump breaks downstream proofs; every PR can introduce a vacuous lemma reviewers wave through. Repair on bump, audit on PR, in CI.
